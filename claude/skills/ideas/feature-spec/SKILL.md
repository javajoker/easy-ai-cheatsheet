# Feature Spec

A focused, single-feature specification for an existing project. The
incremental counterpart to `project-docs`.

## Why this exists

`project-docs` is whole-project. For a new feature in a project that already
has PRD + TECH_DESIGN + a running codebase, regenerating those documents is
wasteful and lossy. But coding without *any* written spec is how features
drift from intent, leak scope, ship without tests, and break implicit
contracts.

A feature spec splits the difference: one tightly scoped doc that
captures intent, the contract delta, and how we'll know it works.

## When to fire

Fire when the user:

- Names a feature to add to an existing project.
- Asks "what would it take to add X?" against a real codebase.
- Wants an RFC or change proposal for a single, bounded change.
- Is staging up for a focused multi-day implementation effort.

Do **not** fire when:

- The project doesn't exist yet — that's `project-prototype` + `project-docs`.
- The change spans the whole architecture — that's `architecture-shepherd`.
- The change is small enough to ship without a spec — go code it.

## Inputs

The skill reads (do not regenerate):

1. `INSTRUCTIONS/projects/<slug>/project-context.md` — stack, conventions,
   test/lint/build commands.
2. `INSTRUCTIONS/projects/<slug>/repository-structure.md` — where things go.
3. Existing PRD + TECH_DESIGN if present (often under `docs/` at the project
   root) — for API contract baseline.
4. The project's knowledge base (`docs/knowledge-base/` if `project-knowledge-base`
   has run) — for entity names + relationships.
5. Project-specific skill references (`skills/projects/<slug>/references/`)
   — for internal API surface.

If the project isn't onboarded yet, **stop and run `project-onboarding` first**
— a feature spec on top of a fuzzy understanding of the project is just
documented guessing.

## Output

One file: `docs/features/FEATURE_<slug>.md` written to the project root (not
to `/mnt/user-data/outputs/`, because this skill targets an existing repo).

Optional delta files when the feature touches them:

- `docs/features/<slug>/api-delta.yaml` — OpenAPI fragment for new/changed
  endpoints.
- `docs/features/<slug>/schema-delta.sql` (or `.prisma`) — DB migration
  preview.
- `docs/features/<slug>/ui-delta.md` — UI surface delta if it changes any
  views.

## Spec structure

The output file follows this fixed shape (see
`references/feature-spec-template.md` for the full template):

1. **Frontmatter** — `feature`, `status` (draft / approved / in-progress /
   shipped), `owners`, `created`, `target_date` (optional), `related_prs`
   (filled later).
2. **Why** — one paragraph. What user problem this solves; what changes for
   the user.
3. **Out of scope** — explicit non-goals. Often longer than the goals.
4. **Load-bearing terms** — produced by `cognitive-alignment`. Each term
   gets a definition and a link to its knowledge-base entity if one exists.
5. **User-facing change** — flow walkthrough, screens affected, copy
   changes, error states.
6. **API contract delta** — endpoint additions/changes with request/response
   examples; explicit "breaking?" yes/no on every change.
7. **Data model delta** — new tables, new columns, indexes, migrations,
   backfill plan.
8. **Background work delta** — new jobs, queue topics, cron entries.
9. **Verification plan** — automated tests (unit / integration / E2E) +
   manual smoke checks + observability (new metrics or logs).
10. **Rollout plan** — feature flag? canary? straight ship? rollback path.
11. **Risks + open questions** — what could break; what isn't decided yet.

## Procedure

### Phase 1 — Anchor against the project

Read the project's `INSTRUCTIONS/projects/<slug>/` and confirm the
feature's stack assumptions match. If it doesn't (e.g. user says "add a
GraphQL endpoint" to a REST-only project), surface the mismatch before
proceeding — don't silently spec a feature that contradicts the platform.

If the project hasn't been onboarded, stop and run `project-onboarding`.

### Phase 2 — Cognitive alignment

Identify the load-bearing terms in the feature description. *"Export-to-CSV"*
seems unambiguous; *"impersonate"*, *"webhook"*, *"discount code"* are not.
Run a `cognitive-alignment` pass and record the terms in section 4 of the
spec.

If existing knowledge-base entities (`docs/knowledge-base/entities/`) cover
those terms, link to them rather than redefining.

### Phase 3 — Contract delta

For each layer the feature touches:

- **API:** name the endpoints. For each: method, path, request body,
  response body, error codes, auth requirements, backwards-compatibility
  status. If a new contract is breaking, the spec must say so explicitly.
- **Data:** name new tables/columns/indexes; preview the migration; specify
  the backfill plan if any existing rows need defaults.
- **UI:** name affected views; sketch the delta (one screen at a time, not
  full mocks).
- **Background:** new jobs, queues, schedules.

For each layer, populate the delta files if the change is non-trivial. A
two-field column addition needs `schema-delta.sql`; a one-string label change
does not.

### Phase 4 — Verification plan

Define how we'll know the feature works:

- **Automated:** which existing test suites must pass; which new tests must
  be added. Specify *level* (unit / integration / E2E) per behaviour.
- **Manual:** smoke checks for things hard to automate (visual layout,
  third-party integration handshakes).
- **Observability:** new metrics, log lines, alerts. If the feature has a
  user-visible failure mode, there must be at least one metric or alert that
  catches it.

Verification plan is non-negotiable. A spec without one is a wish list.

### Phase 5 — Rollout

Decide and record:

- **Feature flag?** Yes if the change is large, risky, or audience-segmented.
  No if it's a small additive change behind existing auth.
- **Canary or straight ship?** Canary if the feature touches a hot path or
  shared infrastructure.
- **Rollback path.** Explicit: "revert PR #N + run migration <down> + flip
  flag X off". If rollback requires data restoration, name the procedure.

### Phase 6 — Risks + open questions

The honest section. List:

- What could break (cite the contract or code path).
- What isn't decided yet (sometimes a spec ships with open questions to
  unblock implementation; those questions get owners and dates).
- Dependencies on other teams or external systems.

## Status vocabulary

Fixed values for the frontmatter `status:` field:

- **draft** — being written; not yet ready for implementation.
- **approved** — accepted; implementation can start.
- **in-progress** — implementation underway; the spec is the source of
  truth for the contract.
- **shipped** — implementation merged + verified; the spec is the
  historical record.

## Companion skills

- `cognitive-alignment` — non-negotiable in Phase 2.
- `project-knowledge-base` — provides the entities the spec links to in
  load-bearing terms.
- `memory-ontology` — records the feature decision + load-bearing terms
  past the session.
- `requirement-audit` — verifies the spec is complete (all 11 sections
  filled, no PARTIAL sections) before status flips to `approved`.

## Companion agents

- `feature-development` — the agent that *consumes* this spec to drive
  the implementation arc (spec → contract lock → code → verify → ship).
- `architecture-shepherd` — invoked if Phase 1 surfaces a feature whose
  shape implies an architectural decision (e.g. "add real-time updates"
  to a polling-based project).

## Anti-patterns

- **Speccing a whole-system change as one feature.** If the spec is
  longer than the existing PRD, you have a project, not a feature — run
  `project-docs` instead.
- **Empty "Out of scope" section.** Scope creep is the default. Naming
  what's *not* in scope is more valuable than naming what is.
- **Skipping Phase 4 (verification).** A spec without a verification plan
  is a wish list and a forecast of a missed regression.
- **No rollback section.** Every feature must have a stated rollback
  path, even if it's "revert the PR — no data changes."
- **Spec divergence during implementation.** When the implementation
  decides differently from the spec, update the spec. The spec is a
  contract, not a one-time prediction.

## Reference files

- `references/feature-spec-template.md` — the full FEATURE_<slug>.md

- `references/api-delta-template.yaml` — OpenAPI fragment template.
- `references/verification-rubric.md` — what makes a verification plan
