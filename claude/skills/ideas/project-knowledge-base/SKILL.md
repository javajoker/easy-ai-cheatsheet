# Project Knowledge Base

A skill that produces a **conceptual** knowledge graph of a software project:
not the file layout, not the task plan, but the domain model and the
relationships between its parts. Think of it as the answer to *"what concepts
does this codebase implement, and how do they relate?"*

## Why this exists

Three artifacts together describe a project end-to-end:

1. **Project context + repository structure** — the project's *shape*.
   Generated by `project-onboarding`.
2. **Task breakdown** — the project's *work plan*. Generated by `task-breakdown`.
3. **Knowledge base** — the project's *concepts and relations*. Generated here.

Most projects skip #3, and pay for it. The cost shows up as: new engineers
wading through the codebase looking for "where does X live?", AI agents
struggling to connect related code across files, and architectural drift
because nobody can see the conceptual structure to know when it's about to
break.

This skill produces #3 in a form that humans, AI agents, and graph tools can
all consume.

## Output

A directory (default `docs/knowledge-base/`) containing:

- `entities/` — one markdown file per significant entity (feature, module,
  service, abstraction, key role). Each file has structured front matter and
  a body that describes the entity and links to where it lives in the code
  and the docs.
- `relations.md` — a manifest of relations between entities. Machine-readable
  block plus a human-readable summary table.
- `terminology.md` — a glossary of project-specific terminology, lifted from
  the cognitive library if one exists from prior sessions on this project.
- `INDEX.md` — entry point. Brief overview, links to every entity file, the
  relation manifest, and the terminology glossary.

The graph manifest in `relations.md` is the load-bearing artifact — tooling
can ingest it directly.

## Entity types

| Type | Examples |
|---|---|
| `feature` | "User authentication", "Payment processing", "Search" |
| `module` | A package, namespace, or directory that owns one or more features |
| `service` | A deployable unit (HTTP server, worker, scheduled job) |
| `abstraction` | A central type or interface that crosses many call sites |
| `external` | An external system the project integrates with (Stripe, S3, Slack) |
| `role` | An actor in the system (User, Admin, Service Account) |
| `decision` | An architectural decision worth recording with rationale |
| `term` | Domain terminology that appears in the codebase or docs |

A project usually has 20–80 entities total in a healthy knowledge base. Many
fewer means too coarse to be useful; many more means too granular.

## Relation types

| Relation | Meaning |
|---|---|
| `implements` | A module implements a feature |
| `uses` | A module/feature depends on another module/feature |
| `exposes` | A service exposes a feature (HTTP, RPC, queue) |
| `belongs-to` | A feature belongs to a larger area |
| `evolves-from` | An entity supersedes or replaces an earlier one |
| `referenced-by` | An entity is mentioned in a doc or test |
| `decided-by` | A decision constrains another entity |
| `integrates-with` | A module integrates with an external system |

All relations are directed. A bidirectional relation appears as two directed
edges (e.g. `A uses B` and `B used-by A`); when generating, write one
direction and let the manifest derive the reverse.

## The six-phase procedure

### Phase 1 — Confirm scope

Ask three questions, max:

- Where should the knowledge base live? Default: `docs/knowledge-base/`.
- Should it cover the entire codebase or a specific area? Default: entire.
- Should existing docs and code comments contribute, or only code structure?
  Default: both.

### Phase 2 — Inventory

Walk the codebase to identify candidate entities:

- Top-level packages or modules → candidate `module` or `service` entities.
- Top-level interfaces or abstract classes → candidate `abstraction` entities.
- Top-level routes and queue consumers → candidate `feature` entries (one
  feature per coherent endpoint cluster).
- External SDK imports → candidate `external` entries.
- README and design docs → mine for `decision`, `feature`, and `term` entries.
- Cognitive library (if present from prior sessions) → import directly as
  `term` entries.

Produce a draft entity list and confirm with the user before writing any
files. The user typically removes 20% (over-granular candidates) and adds
10% (concepts implicit in the code).

### Phase 3 — Write entity files

For each confirmed entity, write `entities/<slug>.md`:

```markdown
---
id: entity-<type>-<slug>-001
title: <Entity Name>
type: feature | module | service | abstraction | external | role | decision | term
status: active | deprecated | planned
created: YYYY-MM-DD
updated: YYYY-MM-DD
related:
  - id: entity-<type>-<other>-001
    relation: implements | uses | exposes | ...
code:
  - <path/to/source.go>
  - <path/to/other.go>
docs:
  - <path/to/design.md>
---

# <Entity Name>

<One-paragraph description in the project's primary language. What this is,
who uses it, why it exists.>

## Where it lives

<List of code files and docs, with one-line descriptions.>

## How it relates

<Plain-text walkthrough of the relations declared in front matter. This is
where a human reader gets the conceptual story; the front matter is for
machines.>

## Open questions

<Anything the skill could not determine — flag for follow-up.>
```

### Phase 4 — Build the relation manifest

Aggregate the `related:` blocks from every entity into `relations.md`:

```markdown
# Relations

## Machine-readable

<relations>
- src: entity-feature-auth-001
  dst: entity-module-auth-001
  relation: implemented-by
- src: entity-feature-auth-001
  dst: entity-feature-rbac-001
  relation: depends-on
... (full list)
</relations>

## Human-readable

| From | Relation | To |
|---|---|---|
| Auth (feature) | implemented-by | auth module |
| Auth (feature) | depends-on | RBAC |
... (one row per directed edge)
```

### Phase 5 — Terminology

Write `terminology.md`. Sources, in priority order:

1. The cognitive library for this project from prior sessions (if any).
2. The `type: term` entities from Phase 3.
3. Frequently-mentioned domain words from README, design docs, and code
   comments.

Each entry:

```markdown
## <Term>

<Definition in the project's primary language. Always include the project-
specific meaning, not a generic dictionary entry.>

**Used in**: <files / features where the term appears>
**Distinct from**: <other terms that might be confused with this one>
```

### Phase 6 — Index and verification

Write `INDEX.md`:

```markdown
# <Project> Knowledge Base

<One-paragraph overview of the project, lifted from project-context.md.>

## Features
- [Auth](entities/feature-auth.md)
- ...

## Modules
- ...

## Services
- ...

## Abstractions
- ...

## External integrations
- ...

## Roles
- ...

## Decisions
- ...

See also:
- [Relations](relations.md)
- [Terminology](terminology.md)
- [Project context](../../INSTRUCTIONS/projects/<slug>/project-context.md)
- [Repository structure](../../INSTRUCTIONS/projects/<slug>/repository-structure.md)
```

Verify:

- Every entity in `INDEX.md` resolves to a file.
- Every relation in `relations.md` resolves on both ends.
- Every code path mentioned in a `code:` field exists.
- Every doc path mentioned in a `docs:` field exists.

Report any unresolved references; ask the user before fixing.

## Companion skills

- `project-onboarding` — should run first. The knowledge base assumes the
  shape-level documentation already exists.
- `cognitive-alignment` — every project-specific term should appear in both
  the cognitive library and the knowledge-base terminology. Reconcile.
- `memory-ontology` — significant decisions discovered during knowledge-base
  generation are good candidates for `type: project` memories. Promote them.
- `task-breakdown` — the knowledge base is a great input for task
  breakdown; tasks reference entities by ID.

## Anti-patterns

- **Mirroring the file system.** A knowledge base is not a file listing.
  Two modules in different directories may belong to the same feature; one
  file may participate in three features. Group by concept, not by path.
- **Over-extracting.** Not every type, function, or file is an entity. A
  good knowledge base has 20–80 entities. If you've generated 200, the
  granularity is wrong.
- **Mis-naming entities for tooling.** Entity IDs are stable; the slug
  becomes a link target. Name conservatively — renames hurt.
- **Writing the bodies in your own voice instead of the project's.** The
  body language follows the project's primary language. The front matter
  is always machine-language (English) regardless.
